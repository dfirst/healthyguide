Именование:
1. Булевые переменные, функции и методы должны начинаться с "is".
2. Функции и методы должны начинаться с глагола.
3. Не использовать and в названиях - это признак нескольких ответственностей.
4. Избегать односимвольных переменных.
5. Давать переменным понятные и развёрнутые имена - буквы не экономим.
6. Числа должны быть вынесены в именованные константы.
7. "Service" - плохое имя. Используйте конкретные роли: "Sender", "Creator", "Reader" и т.д.
8. Названия "core", "utils" - слишком абстрактны, избегать.

Область видимости:
1. Всё, что не используется вне файла или класса - делать приватным ("_name").
2. Неиспользуемые переменные при распаковке обозначать как "_".

Архитектура:
1. В "Django" бизнес-логика должна находиться в "apps/*" или "project_name/*".
2. Класс / функция / метод должны иметь одну ответственность.
3. Если метод “делает A и B” - это кандидат на декомпозицию.
4. Класс должен иметь один публичный метод. Два - повод задуматься.
5. Если приватные методы активно вызывают друг друга - вероятна необходимость декомпозиции.
6. Команды и таски должны быть тонкими - вся логика выносится в бизнес-классы.

Стиль кода:
1. Код должен читаться как линейная история: открыть → обработать → сохранить → отправить
2. Повествование должно быть последовательным и логичным.
3. Единообразие важнее «умных/красивых» решений.

Типизация:
1. Типизация обязательна.
2. Использовать типы осознанно:
    list - последовательность, изменяемая
    tuple - последовательность, неизменяемая
    set - уникальные элементы
    frozenset - неизменяемое множество
3. Optional (int | None) использовать только при реальной необходимости. Получив подтверждение.
4. args / kwargs должны быть типизированы.
5. Предпочитать модели (например, Pydantic) вместо «сырых» dict.

Ошибки и обработка:
1. Следовать принципу "fail fast" - падать рано, а не маскировать ошибки.
2. Не ловить Exception без необходимости.
3. Если ловишь исключение - обрабатывай его явно или пробрасывай дальше.
4. try/except должен быть минимальным и точечным(одна команда):
    except — конкретный
    finally без except — не использовать

Общие принципы:
1. Не реализовывать сценарии «на будущее» без реального запроса.
2. Читаемость и единообразие кода - приоритет.
3. Код должен быть:
    - понятным
    - линейным
    - предсказуемым
    - легко читаемым без контекста
